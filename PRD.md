# PRD — 반도체 시료 생산주문관리 시스템 (SampleOrderSystem)

> 원본 스펙: [개인과제_반도체시료관리.md](./개인과제_반도체시료관리.md)
> 이 문서는 스펙을 실행 가능한 요구사항 형태로 재구성한 것이며, 스펙과 상충하는 경우 원본 스펙이 우선한다.

## 1. 배경 및 목표

가상의 반도체 회사 **S-Semi**는 웨이퍼 공정으로 반도체 시료(Sample)를 생산해 연구소·팹리스·대학 연구실 등에 납품한다. 주문량 급증 이후 엑셀·메모장 기반 관리로 인해 다음 문제가 발생했다:

- 주문 처리 여부를 파악하기 어려움
- 공정 완료 시점을 알 수 없음
- 재고가 충분한데도 불필요한 추가 공정이 발생

**목표**: 콘솔 기반 시스템으로 시료 등록, 주문 접수/승인/거절, 재고에 따른 생산 자동 분기, 생산 큐 처리, 출고, 모니터링을 일관되게 관리한다.

## 2. 사용자 및 역할

| 역할 | 책임 |
|---|---|
| 고객 | 이메일 등으로 시료를 요청 (시스템 외부 액터, 직접 조작하지 않음) |
| 주문 담당자 | 고객 요청을 바탕으로 주문서 작성(시료 예약) |
| 생산 담당자 | 시료 등록, 주문 승인/거절, 생산 현황·재고 관리, 출고 처리 |

콘솔은 담당자(주문/생산 담당자)가 직접 명령을 입력하는 단일 사용자 환경으로 가정한다 (역할별 로그인/권한 분리는 범위 밖).

## 3. 도메인 모델

### 3.1 Sample (시료)

| 필드 | 설명 |
|---|---|
| sample_id | 시료 고유 ID |
| name | 시료 이름 |
| avg_production_time | 평균 생산시간 |
| yield_rate | 수율 = 정상 생산 시료 수 / 총 생산 시료 수 (예: 0.9) |
| stock_quantity | 현재 재고 수량 |

등록된 시료만 주문 가능하다.

### 3.2 Order (주문)

| 필드 | 설명 |
|---|---|
| order_id | 주문 고유 ID |
| sample_id | 대상 시료 ID |
| customer_name | 고객명 |
| quantity | 주문 수량 |
| status | 주문 상태 (아래 3.3 참조) |

### 3.3 주문 상태 흐름 (State Machine)

```
RESERVED --(거절)--> REJECTED
RESERVED --(승인, 재고 충분)--> CONFIRMED --(출고)--> RELEASE
RESERVED --(승인, 재고 부족)--> PRODUCING --(생산 완료)--> CONFIRMED --(출고)--> RELEASE
```

| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수 |
| REJECTED | 주문 거절 (정상 흐름 제외, 모니터링에서 제외되는 종단 상태) |
| PRODUCING | 승인 완료, 재고 부족으로 생산 중 |
| CONFIRMED | 승인 완료, 출고 대기 중 |
| RELEASE | 출고 완료 (종단 상태) |

상태 전이는 위 화살표 방향으로만 허용하며 역방향 전이는 없다.

### 3.4 ProductionQueue (생산 큐)

- PRODUCING 상태로 전환된 주문이 등록되는 대기열
- **생산 라인은 1개**로 고정한다 (스펙: "하나의 생산 라인은 시료를 하나씩 생산"). 즉 한 시점에 실제로 생산이 진행 중인 주문은 최대 1건이며, 나머지 PRODUCING 주문은 큐에서 FIFO로 순서를 기다린다. 여러 생산 라인의 병렬 생산은 다루지 않는다.
- 스케줄링 전략: **FIFO**
- 큐 항목별 실 생산량/총 생산 시간 계산 규칙은 4.5 참조

## 4. 기능 요구사항

### 4.1 메인 메뉴

| 메뉴 | 설명 |
|---|---|
| 시료 관리 | 시료 등록 / 목록 조회 / 이름 검색 |
| 주문 (접수/승인/거절) | 고객 주문 접수, 생산 담당자의 승인·거절 |
| 모니터링 | 상태별 주문 수, 시료별 재고 현황 |
| 출고 처리 | CONFIRMED 주문에 대한 출고 실행 |
| 생산 라인 | 생산 중 시료 현황, 대기 큐 확인 |

메인 화면 진입 시 전체 시료 요약 정보를 함께 표시한다.

### 4.2 시료 관리

- **시료 등록**: name, avg_production_time, yield_rate 입력. `sample_id`는 시스템이 자동 증가로 부여(사용자 입력 없음). `stock_quantity`는 입력받지 않고 0으로 시작하며, 이후 생산라인을 통해서만 증가한다.
- **시료 조회**: 전체 시료 목록 + 현재 재고 수량 표시.
- **시료 검색**: 이름 등 속성 기반 검색.

### 4.3 주문 접수

- 입력값: sample_id, customer_name, quantity. `order_id`는 시스템이 자동 증가로 부여(사용자 입력 없음).
- 대상 sample_id가 등록되지 않은 경우 주문 생성 거부(입력 검증).
- 생성 시 상태는 `RESERVED`.

### 4.4 주문 승인/거절

- RESERVED 목록을 조회하고 특정 주문을 선택해 승인 또는 거절.
- **승인**:
  - 재고 충분 (stock_quantity ≥ quantity) → 즉시 `CONFIRMED`로 전환하며 **이 시점에 재고를 차감**한다 (`stock_quantity -= quantity`).
  - 재고 부족 → 생산 큐에 등록, 상태 `PRODUCING`으로 전환. **이 시점에 남아있던 재고 전량을 이 주문에 즉시 선점(귀속)시켜 `stock_quantity`를 0으로 만든다** — 재고가 부족한 상황이므로 남은 재고 전부는 이미 이 주문 몫이기 때문. 이 선점이 없으면 같은 시료의 다른 주문이 아직 안 줄어든(겹쳐 보이는) 재고를 다시 "사용 가능"한 것으로 잘못 판단해, 두 생산이 각각 완료될 때 같은 재고를 이중으로 차감해 재고가 음수로 떨어지는 버그가 발생한다 (실제 재현·수정: 2026-07-15, [0014] 참고).
- **재고 차감/선점 원칙**: 재고의 최종 확정 반영은 여전히 **주문 상태가 `CONFIRMED`로 전환되는 시점** 개념과 일치한다 — 즉시 승인 경로는 승인 즉시 확정 차감되고, 생산 경로는 "PRODUCING 전환 시 선점(0으로) → 생산 완료 시 수율 초과분만 재고로 환원"의 2단계로 이루어진다 (4.5 참조). 이 2단계 처리는 같은 시료에 대한 주문들이 겹칠 때 동일 재고를 이중으로 사용하지 못하게 하기 위함이며, 겹치지 않는 단일 주문 케이스에서는 최종 재고 값이 기존 방식과 수학적으로 동일하다. 출고(`RELEASE`) 시점에는 재고를 변경하지 않는다.
- **거절**: 즉시 `REJECTED`로 전환. 이후 어떤 처리도 하지 않음.

### 4.5 생산 라인

- 부족분에 대해 수율/오차를 고려해 생산량을 계산한다.
  - 실 생산량 = `ceil(부족분 / yield_rate)`
  - 총 생산 시간 = `avg_production_time * 실 생산량`
- 생산 큐는 FIFO로 처리한다.
- 생산 완료 시: `stock_quantity`에 `실 생산량 - 부족분`(수율 올림으로 생기는 초과분)만 더하고 상태를 `PRODUCING → CONFIRMED`로 전환한다. 부족분만큼은 이미 승인 시점에 이 주문 몫으로 선점되어 있으므로(4.4 참고) 별도로 다시 차감하지 않는다.
- 생산 현황 표기: 현재 생산 중 항목(주문ID/시료명/고객명/주문수량/부족분/실생산량(목표)/경과·총 시간(분)/남은시간(분)/진행률(%))과 대기 큐(FIFO 순번, 주문ID/시료명/고객명/주문수량/부족분/실생산량/소요시간/예상 시작까지 남은시간) 목록을 표시한다. 완료 시점 판정은 Mock Clock(경과시간 ≥ 총 생산 시간) 기준이며, 실제 sleep 없이 메뉴 진입 시마다 체크한다.

### 4.6 출고 처리

- CONFIRMED 상태 주문만 대상.
- 특정 주문을 선택해 출고 실행 → 상태 `RELEASE`로 전환.

### 4.7 모니터링

- **주문량 확인**: RESERVED/CONFIRMED/PRODUCING/RELEASE 상태별 **건수와 수량 합계**를 표시. REJECTED는 제외.
- **재고량 확인**: 시료별 현재 재고 수량과, 해당 시료에 대한 **RESERVED**(아직 승인되지 않아 재고를 전혀 확보하지 못한 수요) 주문 수량 대비 상태 표기. **CONFIRMED 주문은 대기 수량에 포함하지 않는다** — 4.4의 재고 선점 원칙에 따라 CONFIRMED 시점에 이미 재고에서 확정·차감이 끝난 상태라, 다시 "부족 원인"으로 집계하면 이미 해결된 수요를 중복으로 반영해 오판하게 된다 (2026-07-15 발견·수정, [0016] 참고).
  - 여유: 재고 ≥ 대기 수량 (대기 수량이 0인 경우도 포함 — 대기 물량이 없으면 항상 여유)
  - 부족: 0 < 재고 < 대기 수량
  - 고갈: 재고 == 0 그리고 대기 수량 > 0

## 5. 비기능 요구사항

| 항목 | 요구사항 |
|---|---|
| 실행 방식 | 콘솔(CLI) 기반, 담당자가 직접 명령 입력 |
| 데이터 영속성 | 실제 DB 없이 **JSON 파일**을 저장소로 사용한다. 애플리케이션 재시작 후에도 시료/주문 데이터가 JSON 파일로부터 복원되어야 함 ([PoC2] 참고) |
| 아키텍처 | Model/Controller/View 계층 분리 (`View → Controller → Model` 단방향 의존, [PoC1] 참고) |
| 테스트 | 상태 전이, 생산량 계산, 재고 갱신 등 핵심 도메인 로직에 대한 자동화 테스트 필수 |
| 코드 품질 | Clean Code 원칙 준수 (명확한 네이밍, 단일 책임, 불필요한 추상화 지양) |
| 개발 이력 | 의미 단위로 나뉜 커밋 이력 관리 |
| Dummy Data | `tools/dummy_data_generator.py` — `--samples`/`--orders`/`--reset` 지원. 앱과 동일한 `SampleRepository`/`OrderRepository`를 재사용해 `data/`에 시료·주문을 생성한다. 더미 주문 상태는 `PRODUCING`을 제외한 4가지(RESERVED/REJECTED/CONFIRMED/RELEASE) 중 무작위 — `PRODUCING`은 `production_state.json`의 큐/활성 항목과 반드시 짝을 이뤄야 해서 임의 생성 시 불일치가 나기 때문 ([PoC4] 참고) |
| 모니터링 도구 | `tools/run_monitor.py` — `--interval`/`--once`/`--data-dir` 지원. 앱과 동일한 `MonitoringController`를 재사용해 상태별 건수·수량과 재고 티어를 콘솔에 폴링 표시한다 ([PoC3] 참고) |

## 6. 범위 밖 (Out of Scope)

- 사용자 인증/권한 분리 (역할별 로그인)
- 웹/GUI 인터페이스
- 다중 생산 라인 병렬 처리 (스펙상 "하나의 생산 라인은 시료를 하나씩 생산")
- 실제 DB 연동 — **JSON 파일 저장/처리 방식으로 확정** (예: `data/samples.json`, `data/orders.json`). RDB/NoSQL 등 외부 DB 서버 연동은 하지 않는다.

## 8. 설계 결정 로그

### 8.1 확정된 결정

| 결정 사항 | 결론 | 근거 |
|---|---|---|
| 재고 차감 시점 | 주문이 `CONFIRMED`로 전환되는 시점(즉시 승인 시 또는 생산 완료 시) 개념은 유지하되, 부족 승인 경로는 "PRODUCING 전환 시 남은 재고 전량 선점(0으로) → 생산 완료 시 수율 초과분만 환원"의 2단계로 처리한다(같은 시료의 겹치는 주문이 재고를 이중 소진하는 버그 방지, [0014] 참고). 출고(`RELEASE`) 시점에는 재고 변경 없음. | 사용자 확정 (2026-07-15), 2026-07-15 겹침 버그 발견 후 선점 방식으로 수정 |
| 초기 재고 입력 여부 | 시료 등록 시 `stock_quantity`는 입력받지 않고 항상 0으로 시작. 이후 생산라인을 통해서만 재고가 쌓인다. | 사용자 확정 (2026-07-15) |
| ID 채번 방식 | `sample_id`, `order_id` 모두 시스템이 순차 자동 증가로 부여 (사용자 직접 입력 없음). | 사용자 확정 (2026-07-15) |
| 데이터 저장 방식 | 실제 DB 없이 JSON 파일로 저장/처리 (RDB/NoSQL 등 외부 DB 연동 없음). | 사용자 확정 (2026-07-15) |
| ID 형식 | `sample_id`는 `S0001`, `S0002`...; `order_id`는 `O0001`, `O0002`.... 각 리포지토리가 현재 로드된 키의 숫자 접미사 최댓값+1로 계산(`next_id()`), 별도 카운터 파일 불필요 — 재시작에도 안전. | 사용자 확정 (2026-07-15) |
| 생산 완료 처리 방식 | 수동 트리거가 아니라 **Mock Clock 시스템**으로 처리한다. 실제 sleep/백그라운드 스레드 없이, 메인 메뉴·생산 라인 메뉴 진입 시마다 경과시간을 체크해 완료 조건 충족 시 자동으로 `PRODUCING → CONFIRMED` 전환한다. `Clock`(기본 `SystemClock`, 실제 시간 기준) / `FakeClock`(테스트용, 임의로 시간 진행) 추상화를 둔다. | 사용자 확정 (2026-07-15) |
| 생산시간 단위 | `avg_production_time`/`총 생산 시간`의 단위는 **분(minute)**. | 사용자 확정 (2026-07-15) |
